# Start Here

If you are wondering what GosseOS and KnowledgeOS actually are, this is the short version.

## GosseOS

GosseOS is the **working architecture**.
It is the part that asks:
- what kind of problem is this?
- which mode should be used?
- what kind of output should be built?
- where do risk, governance, and control matter?

In practice, GosseOS is about:
- problem classification
- mode selection
- routing
- build logic
- verification
- controlled execution

## KnowledgeOS

KnowledgeOS is the **memory and extraction layer**.
It preserves what should not get lost after a strong working session.

In practice, KnowledgeOS is about:
- extracting core principles
- storing reusable prompts and frameworks
- separating working ideas from canon candidates
- creating linkable knowledge assets
- reducing repeated rethinking

## The basic idea

Most people use AI like this:

**prompt → answer**

This project uses AI more like this:

**raw input → problem class → mode → artifact → verification → reusable knowledge**

That is the core shift.

## Why this exists

Because many AI interactions are:
- too shallow
- too one-off
- too hype-driven
- too weak on governance
- too hard to reuse later

GosseOS / KnowledgeOS are meant to turn AI work into something more:
- structured
- repeatable
- linkable
- clearer
- more accountable

## Current strong themes

- governance-first AI architecture
- bounded reasoning vs. governed execution
- LLM routing by task class
- voice-first hybrid workflows
- character steering / self-governance
- knowledge extraction and canonization
